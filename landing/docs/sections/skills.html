Skills (AI agents)

ACS ships two skills for AI coding agents — both bundled in the same install. The skills sit under skills/ in the monorepo and are mirrored to Grkmyldz148/acs-skills on every push.

The two skills

  • create-acs-sound — design a single @sound block. Input is a natural-language prompt ("a soft pop that's brighter than the default") or a dropped .wav for reverse-engineering, or both. Emits a calibrated @sound block, optionally renders a WAV preview, validates within tolerance bands.
  • acs-soundscape — design a complete project-specific sound layer for a website. The agent reads the project's README, global CSS, hero JSX, and copy, extracts design tokens (color, type, radius, motion, shadow), picks an archetype (terminal-CRT, editorial-paper, precision-metal, organic-wood, brutalist-raw, crystalline-luxury, arcade-warm, clinical-quiet), authors a coherent family of project-prefixed @sound blocks, and writes the .acs file with selector bindings. Validators reject default-preset binding without rationale, generic naming, and incoherent families — so two different sites can't end up sounding alike.

Install — Claude Code (native plugin)

/plugin marketplace add Grkmyldz148/acs-skills
/plugin install acs-skills@acs-skills

Both skills land in ~/.claude/plugins/ and become invokable as /create-acs-sound and /acs-soundscape immediately. Tab autocomplete shows them. Future skills added to the source repo arrive via /plugin update acs-skills — no re-install.

Install — any agent runtime (npx skills)

For Cursor, Codex, OpenCode, Gemini CLI, GitHub Copilot, Warp, Windsurf, and 50+ other agent runtimes, use the skills CLI from vercel-labs: 

npx skills add Grkmyldz148/acs-skills

The CLI clones the mirror, finds both skills, and shows a multi-select picker — install both at once, or just one. The skill lands in the runtime's conventional path (e.g. .cursor/skills/... for Cursor).

Manual install (fallback)

# clone the source repo
git clone https://github.com/Grkmyldz148/acs.git
# copy whichever skill(s) you want into your agent's skills directory
cp -r acs/skills/create-acs-sound ~/.claude/skills/
cp -r acs/skills/acs-soundscape  ~/.claude/skills/

Architecture — progressive disclosure

Each skill follows the same pattern: SKILL.md is a slim INDEX (~2k tokens) — frontmatter, pipeline skeleton with file pointers, and a one-line-per-rule table. Detailed rule bodies live under rules/*.md. The agent opens each rule with Read at the moment the pipeline reaches a step that needs it, so a typical run loads 4–14 rule files out of 50+. No full-skill dump, no skim-and-confabulate.

Run node src/build.mjs in either skill's folder to rebuild its SKILL.md from the rules. The mirror workflow rebuilds automatically before publishing, so the published index is always in sync even if a contributor forgets to run the build locally.

Pipeline — create-acs-sound

1. pipeline-detect-input    → prompt | audio | both
2. pipeline-pick-base-layer → match event-class tokens (event-click, event-tap, …)
3. pipeline-decide-layering → 1, 2, or 3 layers based on event class
4. pipeline-apply-mood      → adjective tokens mutate source/filter/effects
5. pipeline-emit-and-render → emit @sound, optional WAV preview, validate

Pipeline — acs-soundscape

1. pipeline-inventory          → read README, package.json, globals.css, hero, footer
2. pipeline-extract-tokens     → color → freq band, type → texture, radius → attack,
                                 motion → decay, shadow → room, copy → mood + volume
3. pipeline-archetype          → pick or blend from 8 archetype templates
4. pipeline-identity-statement → 3–4 falsifiable sentences ("sounds like X, not Y")
5. pipeline-author-customs     → 5–10 project-prefixed @sound blocks, declared coherence
6. pipeline-bind               → write .acs file: customs first, selectors next
7. validate-*                  → no-default-binding, distinct-naming, coherence, token-fit

Validators reject the file if it's >40% default-bound, if any preset name lacks a project prefix (or collides with a built-in), if the custom family lacks a declared shared trait, or if any preset's parameters fall outside the token sheet's ranges. So the skill cannot regress to "drop tap-tactile on every button" — the failure mode that prompted the skill's existence.

Authoring your own skill

Each rule is a markdown file with frontmatter declaring its category and impact, plus a body containing the rule's logic and an example. The build script (src/build.mjs) reads src/preamble.md plus the rule frontmatter and produces a slim SKILL.md index. See the skills directory for the full source.
